# 用户洞察知识库 — Agent 维护指南

## 一、项目初衷

这是一个为**游戏研发制作人/产品经理**打造的用户洞察知识库。核心信念：**只有足够了解用户，才能做出真正让用户喜爱的游戏或其他互动产品**。

网站的定位不是"心理学百科全书"，而是**面向技术背景转型制作人的实战知识库**。每张知识卡片都必须能回答一个真实的游戏设计问题，并提供可直接落地的设计指导。

### 目标用户画像

- 资深游戏客户端技术负责人，正在转型为有技术背景的游戏研发制作人/产品经理
- 逻辑思维强，习惯以问题为起点，分析问题→解决问题
- 需要在设计游戏产品、优化用户体验、分析游戏数据等场景中运用用户洞察

### 知识体系覆盖范围

从心理学理论到游戏设计实践，最终形成一个完整的"用户洞察 → 产品决策"知识链路：

```
基础理论（心理学根基）
  → 游戏化映射（理论如何应用到游戏场景）
    → 数据验证（用数据证明理论的有效性）
      → 设计模式（可复用的设计模板）
        → 案例库（真实游戏案例拆解）
```

## 三、知识卡片规范

### 3.1 卡片 Frontmatter 字段

```yaml
---
id: "唯一标识符"           # 必填，英文小写+连字符，如 "sdt"
title: "知识卡片标题"       # 必填，中文标题
module: "模块名称"          # 必填，必须匹配模块表中的名称
tags: ["标签1", "标签2"]   # 必填，至少1个，便于检索
difficulty: "基础"          # 必填，三选一：基础 | 进阶 | 高级
prerequisites: []          # 可选，前置依赖的卡片 id 列表
related: []                # 可选，相关卡片 id 列表
status: "draft"            # 必填，三选一：draft | review | published
author: "agent-name"       # 必填，作者标识
date: "2024-01-01"         # 必填，发布日期，必须用双引号包裹！
---
```

### 3.2 卡片正文结构（可读性优先）

**核心原则：每张卡片应该像一篇有趣的博客文章，而不是教科书。**

必须包含以下模块，顺序固定：

```markdown
## 🎯 这个理论解决什么问题？

> "用一句**反直觉或有冲击力**的话描述这个问题"
>
> 例如：你以为玩家想要更多奖励？错了，他们想要的是"差一点就得到"的感觉。

## 📐 核心理论

**先给一句话结论**，再展开。避免罗列式定义。

> 例如：心流的核心不是"让游戏简单"，而是"让挑战恰好在能力边界"。

## ⚙️ 底层机制

用**类比**解释机制，而不是抽象流程图。

> 例如：心流通道就像 CPU 的性能曲线——频率太低跑不动，太高会降频。

## 🎮 游戏设计应用

**先讲一个真实踩坑故事**，再给设计手段。

> "某手游上线了每日签到系统，连续签到30天送SSR。结果：第3天流失率反升15%。为什么？因为'每天必须上线'变成了压力，而非动力。"

## ✅ 设计检查清单

精简到 3-5 条，每条用**正反对比**格式。
```

**必须包含的部分**（推荐但非强制）：

```markdown
## 📊 数据验证方法
只列最关键的 2-3 个指标。

## 📖 新概念解析
用表格列出核心术语的一句话定义。

## 🔗 延伸阅读
每条附一句话说明为什么要读。

## 💡 个人洞察
有态度的观点：理论局限性、实际踩坑、与其他理论的矛盾、你的判断。

## 🔭 下次玩游戏时观察
给读者一个可执行的观察任务。
```

### 3.3 写作风格要求

| 要求 | 说明 | 示例 |
|---|---|---|
| 反直觉开头 | 用冲击力的问题或结论吸引注意力 | "你以为XX？错了，其实是YY" |
| 类比优先 | 用技术人熟悉的类比解释抽象概念 | "就像CPU的频率曲线" |
| 真实故事 | 用具体游戏案例代替抽象描述 | "某手游上线签到系统，结果..." |
| 有态度 | 敢于表达个人观点和判断 | "我觉得这个理论被过度引用了" |
| 可执行 | 给读者具体的行动建议 | "下次玩游戏时观察..." |
| 避免 | 教科书风格、罗列式定义、大段理论原文 | ❌ "该理论由XX于YY年提出..." |

---

**可选部分**（推荐添加）：

```markdown
## 📊 数据验证方法

列出可用于验证该理论在你游戏中效果的数据指标和分析方法。

## 🔗 延伸阅读

- [相关卡片1](/card/卡片id)
```

### 3.3 命名规范

- 文件名：英文小写，连字符分隔，如 `cognitive-load.md`
- ID：与文件名一致（不含 .md 后缀）
- 标签：中文，2-4 个字，如 `["认知负荷", "UI设计", "信息架构"]`

### ⚠️ 重要：date 字段格式

**date 字段必须使用双引号包裹的字符串格式**，否则会导致构建失败：

```yaml
# ✅ 正确格式
date: "2024-01-01"

# ❌ 错误格式（会导致 YAML 解析问题）
date: 2024-01-01
```

原因：YAML 解析器对不带引号的日期格式处理不一致，会导致构建时类型错误。

## 五、内容填充策略

### 5.1 填充顺序规则

1. **逐模块填充**：一个模块达到 50 篇后，再切换到下一个模块
2. **模块顺序**：按序号 01 → 02 → ... → 08 依次填充
3. **当前起始模块**：01-motivation（动机与需求）

### 5.2 单模块内填充策略

每个模块的内容**必须**从浅入深、渐进式添加：

#### 难度分布（每模块 50 篇）

| 难度 | 篇数 | 占比 | 说明 |
|---|---|---|---|
| 基础 | 20 篇 | 40% | 核心理论、经典模型、入门概念 |
| 进阶 | 20 篇 | 40% | 理论扩展、应用深化、跨领域联系 |
| 高级 | 10 篇 | 20% | 前沿研究、争议性话题、深度案例分析 |

#### 内容渐进式添加规则

```
阶段1（前10篇）：基础核心
  - 每个模块的"必知"理论
  - 建立模块的知识框架
  - 高频引用的基础概念

阶段2（11-25篇）：基础扩展
  - 核心理论的变体和衍生
  - 不同学派的观点
  - 更多游戏应用案例

阶段3（26-40篇）：进阶深化
  - 理论的边界条件和局限性
  - 跨模块的知识关联
  - 数据验证方法

阶段4（41-50篇）：高级专题
  - 前沿研究和争议话题
  - 复杂案例深度分析
  - 实战设计模式
```

### 5.3 依赖关系规则

#### 强制依赖规则

- 如果卡片 A 的 `prerequisites` 包含卡片 B，则**卡片 B 必须先于卡片 A 创建**
- 依赖关系形成有向无环图（DAG），不能有循环依赖

#### 依赖关系检查

在创建新卡片前，必须验证：

```bash
npm run validate
```

校验脚本会检查：
- 所有 `prerequisites` 中引用的卡片 ID 是否已存在
- 所有 `related` 中引用的卡片 ID 是否已存在（警告）
- 卡片格式是否符合规范

#### 依赖关系最佳实践

1. **依赖层级不超过 3 层**：A → B → C，不要 A → B → C → D
2. **每个卡片最多依赖 3 个前置卡片**
3. **相关卡片（related）不强制顺序**，但应确保内容有关联
4. **模块内依赖优先**：尽量先完成模块内依赖，再处理跨模块依赖

## 六、内容质量标准

### 6.1 必须满足的标准

- [ ] 五段式结构完整（问题→理论→机制→应用→验证）
- [ ] 问题部分用引号包裹，是一句完整的、可理解的游戏设计问题
- [ ] 理论部分有明确的学术来源（作者+年份+文献名）
- [ ] 机制部分有清晰的因果链（可用 ASCII 流程图表示）
- [ ] 应用部分用表格呈现，包含具体的游戏案例
- [ ] 检查清单至少 3 条，且可直接用于产品评审
- [ ] 前置依赖关系正确

### 6.2 鼓励但不强制的标准

- [ ] 包含数据验证方法
- [ ] 包含反面案例
- [ ] 包含延伸阅读链接
- [ ] 游戏案例来自真实上线产品
- [ ] 案例包含具体的游戏名称和设计细节

### 6.3 禁止的内容

- ❌ 纯理论堆砌，没有游戏设计应用
- ❌ 没有学术来源的"伪科学"内容
- ❌ 过度简化的"快餐式"知识
- ❌ 复制粘贴的重复内容
- ❌ 与用户洞察/游戏设计无关的内容

## 七、Agent 工作流程

### 7.1 首次启动检查

```bash
# 1. 进入项目目录
cd user-insight-kb

# 2. 安装依赖
npm install

# 3. 校验现有内容
npm run validate

# 4. 构建测试
npm run build
```

### 7.2 内容填充流程

#### 步骤1：确定当前填充位置

检查当前模块的已有卡片数量和内容分布：

```bash
# 统计每个模块的卡片数量
# 检查 01-motivation 目录下的 .md 文件数量
ls src/content/knowledge/01-motivation/*.md | wc -l
```

#### 步骤2：制定本批次填充计划

根据当前进度，确定本次需要填充的卡片：

- 确定难度级别（基础/进阶/高级）
- 确定主题范围
- 检查依赖关系，确保前置卡片已存在
- 列出要创建的卡片 ID 和标题

#### 步骤3：创建知识卡片

```bash
# 1. 复制模板
cp src/content/knowledge/_template.md src/content/knowledge/01-motivation/新卡片id.md

# 2. 编辑卡片内容（按五段式结构）
# 3. 运行校验
npm run validate

# 4. 如果校验通过，构建测试
npm run build
```

#### 步骤4：提交变更

```bash
git add src/content/knowledge/01-motivation/新卡片id.md
git commit -m "feat(01-motivation): 添加 [卡片标题] 知识卡片"
git push
```

### 7.3 批量填充建议

每次填充建议 **3-5 张卡片**，原因：

1. **便于校验**：批量创建后统一校验，效率更高
2. **便于审查**：每次变更量可控，便于发现问题
3. **便于回滚**：问题定位和修复更容易
4. **依赖管理**：可以一次性处理一批有依赖关系的卡片

### 7.4 模块切换流程

当一个模块达到 50 篇时：

```bash
# 1. 确认当前模块已完成
ls src/content/knowledge/01-motivation/*.md | wc -l
# 输出应为 50（不含 _template.md）

# 2. 运行全面校验
npm run validate

# 3. 构建测试
npm run build

# 4. 切换到下一个模块，开始规划
# 例如：从 01-motivation 切换到 02-cognition
```

## 十、常见问题

### Q: 文件名冲突怎么办？

A: 确保每个卡片 ID 全局唯一。如果两个模块有相似主题，使用模块前缀区分，如 `motivation-x.md` vs `cognition-x.md`。

### Q: 依赖的卡片在另一个模块，还没创建怎么办？

A: 优先创建被依赖的卡片。跨模块依赖是允许的，但应尽量减少。

### Q: 发现已有卡片内容有误怎么修？

A: 直接编辑对应 .md 文件，修复后运行 `npm run validate` 确认。

### Q: 如何判断一个主题应该放在哪个模块？

A: 根据该主题**主要解决的问题**所属的模块归类。如果跨多个模块，选择最核心的那个，其他模块通过 `related` 字段交叉引用。

### Q: 卡片数量不够50篇但已经没有新的基础主题了怎么办？

## 十一、自动化内容生成

### 11.1 系统架构

项目支持通过 Windows 定时任务自动生成内容卡片：

```
scripts/
├── auto-content-config.json   ← 配置文件（API Key、模型等）
├── auto-content.cjs           ← 核心生成脚本
├── run-auto-content.bat       ← Windows 批处理入口
└── setup-scheduled-task.ps1   ← 定时任务安装脚本
```

### 11.2 配置步骤

**1. 配置 API Key**

编辑 `scripts/auto-content-config.json`：

```json
{
  "provider": "openai",
  "apiKey": "sk-xxx你的API密钥",
  "model": "gpt-4o",
  "baseUrl": "https://api.openai.com/v1",
  "maxCardsPerRun": 1,
  "targetCardsPerModule": 50,
  "projectRoot": "D:\\mimoSpace\\user-insight-kb"
}
```

**2. 手动测试一次**

```bash
cd D:\mimoSpace\user-insight-kb
node scripts/auto-content.cjs
```

确认能正常生成卡片后，再设置定时任务。

**3. 安装定时任务**

以管理员权限运行 PowerShell：

```powershell
cd D:\mimoSpace\user-insight-kb\scripts
.\setup-scheduled-task.ps1
```

### 11.3 生成规则

脚本自动执行以下流程：

```
1. 检查当前模块进度
2. 确定本次难度（基础/进阶/高级）
3. 读取已有卡片避免重复
4. 调用 AI 生成新卡片
5. 保存到对应模块目录
6. 运行格式校验
7. Git 提交
```

难度分配规则：
- 第 1-20 篇：基础
- 第 21-40 篇：进阶
- 第 41-50 篇：高级

### 11.4 管理定时任务

```powershell
# 查看任务状态
Get-ScheduledTask -TaskName "UserInsightKB-AutoContent"

# 手动触发一次
Start-ScheduledTask -TaskName "UserInsightKB-AutoContent"

# 暂停任务
Disable-ScheduledTask -TaskName "UserInsightKB-AutoContent"

# 恢复任务
Enable-ScheduledTask -TaskName "UserInsightKB-AutoContent"

# 删除任务
Unregister-ScheduledTask -TaskName "UserInsightKB-AutoContent"
```

### 11.5 查看生成日志

每次执行后，脚本会输出：
- 当前模块和进度
- 生成的卡片 ID
- 校验结果
- Git 提交状态

如需更详细的日志，可重定向输出：

```bash
node scripts/auto-content.cjs > logs/auto-content.log 2>&1
```
